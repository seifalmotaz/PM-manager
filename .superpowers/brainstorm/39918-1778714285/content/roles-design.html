Role Enforcement: Where to put the check?

Two approaches for enforcing Admin/Member permissions on existing routers

A

tRPC Middleware (Recommended)

Create an adminProcedure that extends protectedProcedure. Admin-only mutations (delete project, delete sprint, manage members, delete comment) use adminProcedure. The middleware queries workspace_members for the user's role and throws UNAUTHORIZED if they're not admin.

Pros: Clean separation. No service-level changes. Existing procedures just change which procedure factory they use. Easy to audit which endpoints are admin-only.
Cons: Adds a DB query per admin procedure call (can be cached). Slightly more boilerplate.

B

Service-Level Checks

Each service method checks the user's role before executing destructive operations. projectService.deleteProject(id, userId) internally verifies the user is admin of the workspace.

Pros: No new procedure factories. Fine-grained control per operation.
Cons: Scattered enforcement — easy to miss a check. Harder to audit. More code duplication.

Middleware Design (Option A)

// trpc.ts - new procedure factories
export const adminProcedure = protectedProcedure.use(
t.middleware(async ({ ctx, next }) => {
const role = await getMemberRole(ctx.user.id, workspaceId)
if (role !== 'admin' && role !== 'owner') {
throw new TRPCError({ code: 'FORBIDDEN' })
}
return next()
})
)
// Workspace-scoped admin (for project/sprint ops)
// Pass workspaceId via input or derive from entity
